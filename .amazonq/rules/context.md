# Контекст проєкту

## Загальне

* Писати українською.
* В кінці файлу має бути новий рядок.

## PHP: стиль коду

* Використовувати `declare(strict_types=1);`.
* Завжди робити імпорти: щоб, наприклад, було Exception, а не \Exception, DateTime, а не \DateTime тощо.
* Але при цьому `use Doctrine\DBAL\Exception as DbalException`. І взагалі класи з назвою Exception не з core PP якось імпортувати з аліасом.
* json-функції мають викликатися з аргументом, що включає кидання виключень.
* Утримуватися від ультракоротких назв змінних (окрім прикладів як `for ($i = 0; ...)`).
* Але виключення нехай ловляться як `$ex`.
* Використовувати pipe-оператор там, де він має сенс.
* Phpdocs на методи робити в декілька рядків. Згадувати всі виключення. Сортувати їх по абетці.
* Якщо в Phpdoc є згадка хоча б одного аргументу методу - то мають бути всі.
* Утримуватися від масивів, де багато елементів в один рядок.
* Коментарі в коді мають бути англійською.
* Якщо є анонімна функція - по можливості робити її статичною. Але без фанатизму.
* Підключення трейтів має бути в алфавітному порядку.

## PHP: архітектура

* Взаємодія front-end з контролерами: приймати переважно DTO, не Request.
* Використовувати DTO не тільки в контролерах, прокидувати їх далі там, де це має сенс.
* Не можна віддавати Entity (або простий масив) на front-end (або в шаблонізатор), только DTO. В DTO всі параметри мають бути іменованими, також не можна покладатися на індекси масивів. Якщо є потреба - то створювати вкладені DTO.
* Ніколи не створювати DTO в контролері вручну, тільки через мапінг. Інколи припустимо створювати DTO вручну в сервісі (якщо це DTO збирається з декількох різних джерел).
* Код, де ми маємо покладатися на те, що масив має структуру типу `['foo' => int, 'bar' => [], 'baz' => string]` - ненадійний. Там, де треба - переробляти на DTO або окремі змінні.
* Якщо це можливо - нехай контролери автоматично привʼязують до себе Entities, id (чи інші параметри) яких передаються в URL.
* В контролерах ловити специфічні виключення і прокидувати їхній текст помилки на front-end якщо ми самі задаємо цей текст. Для загальних виключень кидати загальний текст помилки.
* В контролерах використовувати rate-limits.

## PHP: безпека та валідація

* Робити trim всьому користувацькому вводу. І взагалі очищати його від потенційно небезпечного вмісту.
* Використовувати вбудовану валідацію в DTO (через Assert) там, де це можливо.
* id, які є integer, мають бути додатними.
* Уникати проблеми N+1 запитів.
* Враховувати потенційні ризики безпеки (включно, але не обмежуючись OWASP TOP 10).

## PHP: тести

* Пишемо end-to-end тести, тобто покриваємо контролери (зрідка можемо покривати й інші класи).
* Тести мають дотримуватися принципів FIRST.
* В тестах покривати всі happy path і прагнути покривати побільше unhappy path.
* В тестах, якщо використовуєш мок, обовʼязково мають бути очікування.
* Не створювати окремі тести, якщо можна додати тест-кейс в існуючий dataProvider.
* По тестовому класу на контролер.

## PHP: робота з базою даних
* Якщо ми в певній сутності оновлюємо поле, яке є колекцією, то замість повного перезатирання колекції ми маємо порівняти її з поточною, і видаляти тільки неактуальні елементи та додавати ті, яких ще немає.

## PHP: міграції

* В міграціях запити мають логічно розбиватися на декілька рядків.
* Якщо створюємо таблицю в рамках поточного коміту - то оновлюємо існуючу міграцію, а не створюємо нову.
* ALTER однієї таблиці - в один запит (якщо можливо).
* По окремому файлу міграції на таблицю (якщо можливо).

## PHP: інструменти

* Команди мають завжди запускатися в docker container.
* Запускати phpunit, phpcs, phpstan без grep/head/tail тощо (бо у разі помилок не доводилося робити повторний запуск, щоб дізнатися, де саме проблема).
* Щоб з консолі виконувати SQL запити - запускати їх через `bin/console dbal:run-sql`.
* Якщо додаємо в YAML-переклади - не забувати згенерувати й JS-переклади.

## Кешування

* Бекенд — Redis через `cache.adapter.redis_tag_aware`, у тестах — `cache.adapter.array`.
* Кешувати тільки DTO, ніколи не Doctrine Entity (detached state, проблеми з серіалізацією).
* Використовувати tag-based інвалідацію через `TagAwareCacheInterface`.
* Кешувати дані, які рідко змінюються або не змінюються після створення (опубліковані турніри, довідники).
* Не кешувати дрібні запити (1-3 записи в таблиці, прості SELECT по PK).
* Не кешувати дані з великою кількістю комбінацій параметрів (низький hit rate).
* TTL як страховка — навіть без інвалідації дані мають оновитися за розумний час.
* Інвалідація має бути каскадною: зміна турніру чистить кеш пов'язаних гравців і команд.
* Складну інвалідацію виносити в `CacheInvalidator`, просту — робити на місці.
* При деплої чистити Redis через `cache:pool:clear cache.app`.
